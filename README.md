# Пайплайн аналитики маркетплейсов с ИИ-классификацией
[![CI](https://github.com/aslan-py/interika/actions/workflows/ci.yml/badge.svg)](https://github.com/aslan-py/interika/actions/workflows/ci.yml)
Консольное приложение для мониторинга конкурентов и поиска нишевых товаров.
Собирает каталог маркетплейса, с помощью нейросети определяет ценовой сегмент
каждого товара и автоматически ставит задачи менеджерам в CRM для проработки
самых интересных позиций.

**Поток данных:**

```
Wildberries API  →  Нормализация  →  LLM-классификация  →  Отбор  →  AmoCRM
   (парсинг)         (Pydantic)        (DeepSeek)         позиций    (задачи)
                                                                        ↓
                                                                     Отчёт
```

## Что умеет

- **Парсинг Wildberries** — асинхронный сбор с пагинацией, обработкой
  rate-limit (429 + экспоненциальный backoff), graceful degradation и
  чекпоинтами (продолжение с прерванной страницы)
- **Нормализация** — приведение к единой Pydantic-схеме `Product`,
  дедупликация по id, отбраковка битых записей
- **ИИ-классификация** — сегменты Эконом / Стандарт / Премиум через DeepSeek;
  батчинг, дедупликация по названию и Redis-кэш кратно снижают число запросов;
  ценовой fallback при недоступности LLM
- **Интеграция с AmoCRM** — создание задач через API; трекинг отправленных
  позиций в Redis (повторный запуск берёт следующие, а не дубли); fallback в
  JSON при отсутствии токена
- **Отчёт** — сводка о прогоне (собрано / сегменты / задачи) в JSON и лог
- **Гибкий CLI** — запуск целиком или по этапам, выбор категории, режим
  отладки правил классификации без кэша

## Запуск локально (на ПК)

Требуется **Python 3.12+** и **Docker** (только для Redis — сам Python-код
запускается на хосте).

**1. Клонировать репозиторий**

```bash
git clone <url-репозитория>
cd interika
```

**2. Создать виртуальное окружение и установить зависимости**

Есть два способа — выберите **один**. Быстрее и надёжнее через
[**uv**](https://docs.astral.sh/uv/) (проект уже содержит `pyproject.toml`
и `uv.lock` с зафиксированными версиями), но подойдёт и обычный `pip`.

<details open>
<summary><b>Вариант 1 — через uv (рекомендуется)</b></summary>

`uv` сам создаст `.venv`, поставит нужную версию Python и все зависимости
строго по `uv.lock`:

```bash
# Установить uv, если его ещё нет
#   Windows (PowerShell):
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
#   Linux / macOS:
curl -LsSf https://astral.sh/uv/install.sh | sh

# Создать окружение и поставить зависимости (одна команда)
uv sync
```

После этого команды запускаются через `uv run` — активировать окружение
вручную не нужно (например, `uv run main.py`, `uv run pytest`).

</details>

<details>
<summary><b>Вариант 2 — через pip</b></summary>

Зависимости продублированы в `requirements.txt`. Ставим их в отдельное
виртуальное окружение, чтобы не засорять систему:

```bash
# Создать виртуальное окружение
python -m venv .venv

# Активировать его
#   Windows (PowerShell):
.venv\Scripts\Activate.ps1
#   Linux / macOS:
source .venv/bin/activate

# Установить все зависимости
pip install -r requirements.txt
```

</details>

**3. Создать файл `.env`**

Копируем шаблон `.env.example` в `.env`:

```bash
# Windows (PowerShell):
copy .env.example .env
# Linux / macOS:
cp .env.example .env
```

Затем открываем `.env` в редакторе и заполняем **обязательные** поля:

| Переменная | Что вписать |
|-----------|-------------|
| `DEEPSEEK_API_KEY` | ключ доступа к DeepSeek (для ИИ-классификации) |
| `AMOCRM_ACCESS_TOKEN` | долгоживущий токен интеграции AmoCRM |
| `AMOCRM_SUBDOMAIN` | поддомен аккаунта AmoCRM (например, `mycompany`) |
| `WB_QUERY` | категория для парсинга — по умолчанию `смартфон`; поменяйте, если нужна другая категория |

Остальные параметры уже заполнены разумными значениями по умолчанию
(полное описание — в разделе «Конфигурация»). Файл `.env` в `.gitignore` —
секреты не попадают в git.

**4. Запустить Redis и панель RedisInsight**

Приложению для кэша нужен Redis. Отдельно поднимать его вручную не надо —
обе службы (сам **Redis** и веб-панель **RedisInsight** для просмотра
содержимого кэша) поднимаются одной командой из `docker-compose.yml`:

```bash
docker compose up -d redis redisinsight
```

- Redis слушает на `localhost:6379` (адрес уже прописан в `.env`).
- Панель RedisInsight открывается в браузере по адресу
  **http://localhost:5540** — там можно вручную просматривать ключи кэша
  сегментов и трекинг отправленных в CRM задач.  
  (На фронтовой части: Use recommended settings + I have read and understood the Terms -> Connect existing database -> Connection Setings ->поле Host меняем на redis -> test connection -> add redis database)

**5. Проверить, что всё работает**

```bash
# через uv:
uv run pytest tests/ -q     # тесты (не требуют ключей)
uv run ruff check .         # линт

# либо с активированным venv (pip-вариант):
pytest tests/ -q
ruff check .
```

## Конфигурация (`.env`)

| Переменная | Назначение | Обязательна |
|-----------|-----------|-------------|
| `WB_QUERY` | категория/поисковый запрос Wildberries | да |
| `WB_MAX_PRODUCTS` | сколько товаров собирать при полном прогоне | нет (10000) |
| `DEEPSEEK_API_KEY` | ключ DeepSeek для классификации | да* |
| `DEEPSEEK_MODEL` | модель (`deepseek-chat`) | нет |
| `LLM_BATCH_SIZE` | товаров в одном запросе к LLM | нет (30) |
| `SEGMENT_P_ECONOMY` / `SEGMENT_P_PREMIUM` | перцентили ценового fallback | нет |
| `REDIS_URL` | адрес Redis (`redis://localhost:6379/0`) | нет |
| `AMOCRM_SUBDOMAIN` | поддомен аккаунта AmoCRM | да** |
| `AMOCRM_ACCESS_TOKEN` | долгосрочный токен интеграции | да** |
| `DEMO_MODE` | `true` — собирать `DEMO_LIMIT` товаров вместо всех | нет |
| `DEMO_LIMIT` | лимит товаров в демо-режиме | нет (50) |

\* Без `DEEPSEEK_API_KEY` классификация переключается на ценовой fallback.
\*\* Без токена AmoCRM задачи сохраняются в `output/tasks_*.json`.

Тестам ключи **не нужны** — они мокают внешние сервисы.

## Запуск

Приложение можно запускать двумя способами — **локально** (Python на хосте)
или **целиком в Docker**. И в том, и в другом случае можно прогнать весь
пайплайн разом либо отдельные этапы.

> 💡 **Демо-режим.** Если в `.env` стоит `DEMO_MODE=True`, парсер собирает не
> все товары, а только `DEMO_LIMIT` штук (по умолчанию 50). Это удобно для
> быстрой проверки: полный прогон на 10 000 товаров занимает заметно больше
> времени и расходует баланс LLM. Для боевого прогона поставьте
> `DEMO_MODE=False`.

### Вариант A — локально

Redis поднят (см. «Запуск локально»). Полный пайплайн
(парсинг → нормализация → LLM → CRM → отчёт):

```bash
# через uv (окружение активировать не нужно):
uv run main.py

# либо с активированным venv (pip-вариант):
python main.py
```

> Дальше в примерах используется `python main.py`. Если ставили через uv —
> подставляйте `uv run main.py` (либо `uv run python main.py`); все флаги
> работают одинаково.

### Вариант B — в Docker

Одна команда собирает образ приложения и поднимает вместе с ним Redis и
RedisInsight, после чего прогоняет весь пайплайн:

```bash
docker compose up --build
# либо скрытно, без логов в консоли (можно смотреть в logs/total.log)
docker compose up --build -d
# Если не получится спарсить, все попытки скинут ошибку 429, повторно запустить пайплайн
docker compose restart app
# или
docker compose up -d app
```

Результаты появляются на хосте в папке `output/`, логи — в `logs/`
(проброшены через volume). Здесь отдельно поднимать Redis не нужно —
`docker-compose.yml` поднимает всё сам.

**Запуск этапов и отдельных команд в Docker**

Контейнер `app` по умолчанию прогоняет весь пайплайн и завершается, а внутри
образа уже установлен `uv`. Чтобы запустить конкретный этап или свою команду,
есть два пути:

```bash
# 1. docker compose run — стартует свежий одноразовый контейнер и удаляет его
#    после выхода (--rm). Надёжный способ для разовых прогонов по этапам,
#    контейнер держать запущенным не нужно. Redis поднимется сам (depends_on).
docker compose run --rm app uv run --no-dev python main.py --parsing
docker compose run --rm app uv run --no-dev python main.py --normalizer --llm
docker compose run --rm app uv run --no-dev python main.py --reset

# 2. docker compose exec — выполняет команду ВНУТРИ уже запущенного контейнера
#    app (например, если подняли его через `up -d` и он ещё работает).
docker compose exec app uv run --no-dev python main.py --llm --no-cache
docker compose exec app uv run --no-dev python main.py --crm --category смартфоны

# Открыть shell внутри контейнера и работать вручную:
docker compose exec app bash          # если контейнер запущен
docker compose run --rm app bash      # свежий контейнер под ручные команды
```

> ⚠️ `exec` работает только пока контейнер `app` **запущен**. Так как после
> прогона пайплайна он выходит, для разовых команд по этапам удобнее
> `docker compose run --rm app …`.

### Запуск по этапам

Пайплайн разбит на 4 этапа, их можно запускать по отдельности или цепочкой.
Порядок флагов не важен — последовательность задаёт сам код:

| Флаг | Этап |
|------|------|
| `--parsing` | парсинг Wildberries → `output/wb_raw_*.json` |
| `--normalizer` | нормализация в `Product` → `output/normalized_*.json` |
| `--llm` | ИИ-классификация сегментов → `output/classified_*.json` |
| `--crm` | отбор позиций и задачи в AmoCRM |

```bash
python main.py --parsing                     # только парсинг
python main.py --parsing --normalizer --llm  # до классификации, без CRM
python main.py --normalizer --llm --crm      # с середины до конца
python main.py --llm                          # взять последний normalized → LLM
```

Смежные выбранные этапы передают данные друг другу в памяти. Если этап
запущен в одиночку (предыдущий пропущен), данные берутся с диска — из
последнего файла соответствующего типа в `output/` (`--llm` берёт свежий
`normalized_*.json`, `--crm` — свежий `classified_*.json`; какой именно тип
файла соответствует этапу — см. таблицу в разделе «Результаты прогона»).

### Управляющие флаги

| Флаг | Что делает |
|------|-----------|
| `--fresh` | парсинг с нуля, игнорируя checkpoint (только вместе с `--parsing`) |
| `--reset` | очистить `output/` и кэш Redis (выполняется до этапов) |
| `--no-cache` | не использовать Redis-кэш сегментов — переклассифицировать всё заново (после правки правил классификации) |
| `--category <имя>` | какой файл брать с диска на отдельном этапе, напр. `--crm --category смартфоны` |

```bash
python main.py --reset                 # чистка перед прогоном с нуля
python main.py --parsing --fresh       # парсинг заново, без checkpoint
python main.py --llm --no-cache        # переклассификация без кэша
```

### Результаты прогона

После полного прогона в `output/` лежат файлы каждого этапа. Имя файла — это
`{тип}_{категория}_{дата}_{время}.json`, поэтому разные категории и разные
прогоны не перезаписывают друг друга, а «последний» файл каждого типа
определяется по времени:

| Файл | Что это | Кто создаёт |
|------|---------|-------------|
| `wb_raw_<категория>_<время>.json` | **сырой** ответ Wildberries как есть, до обработки | `--parsing` |
| `normalized_<категория>_<время>.json` | **нормализованные** товары в единой Pydantic-схеме `Product` (дедуп, битые записи убраны), поле `segment` ещё пустое | `--normalizer` |
| `classified_<категория>_<время>.json` | те же `Product`, но **после LLM** — с проставленным сегментом (Эконом / Стандарт / Премиум) | `--llm` |
| `report_<категория>_<время>.json` | **итоговый отчёт** о прогоне: сколько собрано, распределение по сегментам, сколько создано задач | `--crm` (конец пайплайна) |
| `tasks_<время>.json` | **задачи для CRM** — сохраняются сюда только как fallback, если не задан токен AmoCRM (иначе задачи уходят прямо в AmoCRM) | `--crm` |

Та же сводка из `report_*.json` дублируется в лог. Созданные задачи видны в
интерфейсе AmoCRM; без токена они сохраняются в `output/tasks_*.json`.
