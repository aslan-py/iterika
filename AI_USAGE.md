### Сессия 7 — Исследование лимитов WB и стратегия сбора 10 000 товаров (2026-07-12)

**Модель:** Claude Sonnet 4.6 (Claude Code)

**Итог сессии:** исследовали причины непостоянного количества страниц от WB (12 → 17 → 32), выяснили природу ограничений публичного search API, изучили антибот-защиту WB через `x-pow`, рассмотрели варианты достижения 10 000 товаров. Попытка переключиться на каталожный API (`catalog.wb.ru`) не дала результата — эндпоинт нестабилен без токена.

**Промпты:**
- **Prompt 1** — «Сейчас я попробовал скачать 10 000 товаров, на 10-й странице всё оборвалось. Если запущу второй раз, начнёт сначала, а не с того места где остановились.»
- **Prompt 2** — «Наверное, надо другой каталог по категории» — попытка переключиться с search API на catalog API (`catalog.wb.ru`)
- **Prompt 3** — «Не сработало» / «404» / «502» — серия отладки каталожного эндпоинта
- **Prompt 4** — «Откатывай назад, как было до этого» — отмена самостоятельного решения использовать мульти-запросный подход без согласования
- **Prompt 5** — «Сейчас дошёл до 32 страниц. Что мне нужно сделать чтобы продолжить? В output всё корректно?»
- **Prompt 6** — [Stack Overflow ссылка про x-pow] «Это поможет оптимизировать запрос?»
- **Prompt 7** — «Если надо 10 000 наименований — как это сделать? Тяжело ли получить x-pow токен?»

**Что выяснено:**
- WB search API ограничен ~3200 товарами на один запрос (32 страницы × 100) — потолок меняется в зависимости от времени суток и CDN-узла
- Непостоянство (12 / 17 / 32 страницы) объясняется антибот-системой WB: без заголовка `x-pow` сервер отдаёт урезанную выдачу
- `x-pow` токен генерируется JS на странице WB, живёт ~4 минуты, извлекается через Playwright/Puppeteer из `localStorage['session-pow-token']`
- Каталожный API (`catalog.wb.ru`) нестабилен без токена — возвращал 404/502 в ходе тестов
- Для набора 10 000 товаров рассмотрены два варианта: (1) разные параметры сортировки одного запроса с дедубликацией по ID, (2) x-pow токен через Playwright

**Архитектурное решение (в процессе выбора):**
- Вариант 1 (разные `sort=`) — проще, без тяжёлых зависимостей
- Вариант 2 (x-pow + Playwright) — правильнее, но +130 МБ Chromium и управление токеном
- Параллельно тестируется запрос «кроссовки» для проверки глубины выдачи по другой категории

**Модель:** Claude Sonnet 4.6 (Claude Code)

- **Prompt 4** — «Что-то прям совсем мало нашлось: 1558. Как-то странно. Давай с телефонами попробуем.»
- **Prompt 5** — «Нет, получилось всего 441. Почему так мало?»
- **Prompt 6** — «Может, попробуем Ozon? Есть ли аналогичная ссылка?»
- **Prompt 7** — [после 403 от Ozon] «Я всё откатил назад. Хотел бы поправить только checkpoint — чтобы не удалял при пустой странице.»
- **Prompt 8** — «Другая нейронка предложила priceU-фильтр — мы это пробовали?»
- **Prompt 9** — «Дедупликация работает по магазину или по названию товара?»
- **Prompt 10** — «Сохрани промпты и историю попыток обойти ограничение WB.»

**Что попробовали и почему не сработало:**

**Попытка 1 — Мультисортировка (popular / rate / priceup / pricedown)**
- Идея: каждая сортировка даёт свой пул товаров → суммарно 4 × 3200 = ~12 800
- Результат: WB возвращает одинаковые 300 nm_id при любой сортировке — каталог один, порядок разный
- Вывод: `sort=` параметр не меняет состав выдачи, только порядок

**Попытка 2 — Ценовые диапазоны через параметр `priceU`**
- Идея: разбить каталог на непересекающиеся ценовые срезы (0–1000 руб, 1000–3000 руб, …)
- Результат: все диапазоны вернули те же самые nm_id (`+0 новых из 100`). Лог:
  ```
  [0–1000 руб] Стр 4: +0 новых из 100
  [1000–3000 руб] Стр 1: +0 новых из 100
  [3000–6000 руб] Стр 1: +0 новых из 100
  ```
- Вывод: `exactmatch` API игнорирует параметр `priceU` — он работает только в каталожном API

**Попытка 3 — Несколько поисковых запросов (смартфон, телефон, xiaomi, samsung…)**
- Идея: каждый запрос даёт свой срез каталога
- Результат: 5 запросов дали 985 уникальных товаров, 21 запрос — ~1558. Запрос «смартфон samsung» оборвался на 2-й странице из-за 429
- Вывод: частично работает, но каждый запрос упирается в те же ~300 уникальных; масштабируется плохо

**Попытка 4 — Переключение на Ozon**
- Идея: Ozon поддерживает настоящую категорийную пагинацию без циклов
- Эндпоинт: `https://www.ozon.ru/api/composer-api.bx/page/json/v2?url=/category/smartfony-15502/?page=N`
- Результат: HTTP 403 Forbidden — Ozon требует сессионные куки (JavaScript challenge), простой httpx-запрос блокируется антиботом
- Вывод: для работы с Ozon нужен Playwright/Selenium, что выходит за рамки текущего скоупа

**Ключевое открытие о «3200 товарах»:**
Предыдущий результат 3200 был получен **без дедупликации** — парсер добавлял все товары с каждой страницы, включая повторы с 4-й страницы. Реально уникальных nm_id было только ~300. С дедупликацией потолок — ~300 уникальных на один запрос.

**Прояснение про дедупликацию по nm_id:**
- Дедупликация работает по `id` товара (nm_id — уникальный идентификатор листинга)
- Samsung A100 у продавца №1 и Samsung A100 у продавца №2 имеют разные nm_id → оба попадают в выгрузку (корректно)
- Отсекался именно один и тот же листинг одного продавца, если WB возвращал его повторно на 4-й+ странице

**Итоговое архитектурное решение:**
- Оставлен оригинальный WB парсер без дедупликации — собирает ~3200 записей (включая дубли после 3-й страницы)
- Checkpoint больше не удаляется при пустой странице — можно продолжить сбор при следующем запуске
- Ограничение задокументировано; по ТЗ упрощённый скоуп (500–1000 товаров) засчитывается, ~3200 записей превышает этот порог
- Ozon оставлен как потенциальный источник (файл `src/parser/ozon.py` создан и сохранён) на случай если понадобится настоящая пагинация

- `main.py` — `run_stages(args)`: выполняет выбранные этапы по порядку пайплайна, смежные передают данные в памяти, разрыв → загрузка с диска
- `main.py` — `_dispatch()`: reset (первым) → набор этапов → или полный пайплайн; предупреждение о `--fresh` без `--parsing`

- Всего тестов: 84
- ROADMAP.md — таблица команд дополнена `--reset` и примерами комбинаций

**Архитектурные решения:**
- **Комбинирование через флаг ran_previous** — передача в памяти только между смежными выполненными этапами; разрыв сбрасывает флаг → следующий этап читает файл. Просто и покрывает все комбинации
- **Порядок задаёт код, не ввод** — устойчивость к «неправильному» порядку флагов бесплатно, т.к. argparse не хранит последовательность
- **--reset выполняется до этапов** — `--reset --parsing --llm` = чистый прогон одной командой; reset не в группе с этапами, комбинируется свободно

### Сессия 15 — Мультикатегория: категория в имени файла + --category (2026-07-12)

**Модель:** Claude Opus 4.8 (Claude Code)

**Итог сессии:** решена проблема нескольких категорий. Раньше при парсинге телефонов, кроссовок и удочек CRM брала только последний `classified` по дате — остальные игнорировались. Теперь имена файлов включают категорию (`classified_смартфоны_<время>.json`), а флаг `--category` выбирает нужную. Формат времени дополнен секундами — нет коллизий при быстром парсинге.

**Промпты:**
- **Prompt 1** — «Если я спарсю несколько категорий (телефоны, кроссовки, удочки), будет несколько classified-файлов. Как CRM выберет какой? Он берёт последний по дате? А если я хочу отдельно поставить задачи, у меня несколько classified — он просто последний смотрит?»

- **Prompt 3** — «Запиши сессию, добавь в ROADMAP пояснение, закоммить»

**Разобранный вопрос:** до этой сессии `load_latest_products('classified')` брал последний файл по имени (= по дате), т.е. только одну категорию — классифицированную последней. Остальные категории CRM не видела. Плюс формат имени без секунд → две категории в одну минуту перезаписывали друг друга

**Что сделано:**
- `src/storage.py` — `slugify(text)` (категория → безопасное имя, кириллица сохраняется, пробелы → `_`), `build_filename(prefix, category)` → `{prefix}_{slug}_{время-с-секундами}.json`
- `src/storage.py` — `_latest_file(prefix, category='')`: фильтр по категории (`{prefix}_{slug}_*`) или последний любой; **сортировка по mtime** вместо имени (категория в имени сбивала бы алфавитный порядок)
- `load_latest_raw(category)`, `load_latest_products(prefix, category)` — принимают категорию
- `src/parser/wb.py` — `save_raw(products, category)` через `build_filename`; убран неиспользуемый импорт `datetime`
- `src/normalizer/normalize.py` — `save_normalized(..., category)`; категория берётся из товаров (`products[0].category`) или `settings.wb_query`
- `main.py` — этапы `stage_normalize/llm/crm` принимают `category`; аргумент `--category` прокинут через `run_stages`
- `tests/test_storage.py` (новый) — 7 тестов: slugify, build_filename, загрузка по категории / без / отсутствующей
- Всего тестов: 92

**Живая проверка:**
```

--category смартфоны → найдено 300 товаров, категория смартфоны
--category удочки     → НЕТ (правильно)
без категории         → последний по времени
```

**Архитектурные решения:**
- **Категория в имени, а не только внутри файла** — позволяет адресную загрузку без чтения содержимого; каждая категория живёт своим файлом и своим пулом задач в Redis
- **Сортировка по mtime** — при категории в имени лексикографический порядок ≠ хронологический (`кроссовки` < `смартфоны`), поэтому «последний» определяется по времени модификации файла
- **Секунды в метке времени** — устраняют коллизию имён при парсинге нескольких категорий подряд в пределах минуты
- **Обратная совместимость** — `category=''` сохраняет прежнее поведение (последний любой файл), старые файлы без категории в имени по-прежнему читаются

### Сессия 16 — Флаг --no-cache для отладки правил + вынос промптов (2026-07-12)

### Сессия 17 — Бренд-центричный промпт: убрали ценовые границы (2026-07-12)

**Что сделано:**

**Живая проверка (смартфоны, 300 товаров, --no-cache):**

- `.env` / `.env.example` — `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` с пояснением про GitHub Secrets
- Весь проект: `ruff check .` — All checks passed; 100 тестов зелёные

**Docker (Этап 7 — возвращён app-сервис):**
- `docker-compose.yml` — добавлен сервис `app` (build из Dockerfile, `env_file: .env`, `REDIS_URL=redis://redis:6379/0` override, depends_on redis healthy, volumes output+logs)
- Redis торчит портом `6379:6379` наружу → доступен и app-контейнеру, и локальному коду с ПК одновременно
- `Dockerfile` — CMD `uv run --no-dev python main.py` (полный пайплайн)
- три сценария запуска: (1) локально + Redis в Docker, (2) `docker compose up --build` автопрогон, (3) `docker compose run --rm app uv run main.py --флаги` поэтапно
- `.env` редактируется на ПК до запуска; Docker подхватывает через `env_file` — внутрь контейнера заходить не надо
- `.github/workflows/ci.yml` — job `docker-build`: `docker build` (проверка сборки образа, без запуска пайплайна — в CI нет секретов и смысла парсить WB)

**Архитектурные решения:**
- **ruff + pytest, а не только ruff** — линт и тесты проверяют ортогональные свойства (чистота vs работоспособность)
- **Telegram-токен в Secrets, не в .env** — CI не имеет доступа к локальному `.env`; секреты CI — только через GitHub Secrets
- **continue-on-error на уведомлении** — если Secrets не заданы, CI не падает из-за шага нотификации; сам линт/тесты остаются обязательными
- **Долг закрыт перед CI** — иначе первый же прогон CI был бы красным на линте старых файлов
- **app-сервис с автозапуском + docker compose run для этапов** — `up` даёт демо «одной кнопкой», `run` даёт доступ ко всему CLI внутри контейнера; ничего дополнительно кодить не нужно
- **CI собирает образ, но не запускает** — сборка проверяет корректность Dockerfile; запуск пайплайна требует секретов и сети, это дело локального демо
